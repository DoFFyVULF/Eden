2.3. Программная реализация серверной части (Backend)

2.3.1. Структура проекта и настройка окружения

Серверная часть автоматизированной системы управления салоном красоты реализована с применением фреймворка NestJS версии 11, функционирующего в среде выполнения Node.js. Выбор данного фреймворка обусловлен его модульной архитектурой, встроенной поддержкой паттернов проектирования, а также возможностью использования строгой типизации посредством языка TypeScript.

Структура проекта организована по модульному принципу, при котором каждый функциональный блок инкапсулирован в отдельном модуле NestJS. Корневой модуль приложения (AppModule) осуществляет регистрацию всех зависимостей и объединяет функциональные модули системы.

Файл main.ts выступает точкой входа в приложение и выполняет следующие функции:
— инициализация NestJS-приложения с настройками логирования;
— конфигурирование CORS (Cross-Origin Resource Sharing) для разрешения кросс-доменных запросов с клиентского приложения;
— подключение middleware для обработки HTTP-cookie, что необходимо для работы с refresh-токенами;
— настройка глобальной валидации входящих данных посредством ValidationPipe с параметрами whitelist и forbidNonWhitelisted, обеспечивающими отсечение неизвестных свойств;
— выполнение скрипта инициализации учётной записи администратора при первом запуске системы;
— запуск HTTP-сервера на конфигурируемом порту (по умолчанию — 4200).

Конфигурация приложения вынесена в переменные окружения, что соответствует принципам методологии Twelve-Factor App. К конфигурируемым параметрам относятся: строка подключения к базе данных PostgreSQL, секретные ключи для подписи JWT-токенов, настройки CORS и переменные окружения для различных режимов работы (разработка, промышленная эксплуатация).

В приложении реализована глобальная валидация входных данных на основе декораторов библиотеки class-validator. Каждый запрос, содержащий тело (POST, PATCH), проходит автоматическую проверку на соответствие DTO-классам (Data Transfer Object), что обеспечивает защиту от некорректных данных и снижает вероятность возникновения ошибок времени выполнения.

2.3.2. Реализация модуля аутентификации и безопасности

Модуль аутентификации (AuthModule) представляет собой ключевой компонент системы обеспечения безопасности разрабатываемого приложения. Модуль отвечает за регистрацию пользователей, аутентификацию, авторизацию и управление сессиями с применением JWT-токенов (JSON Web Token).

Архитектура модуля построена на основе паттерна «Контроллер–Сервис» и включает следующие компоненты:

— AuthController — контроллер, обрабатывающий HTTP-запросы к эндпоинтам аутентификации;
— AuthService — сервис, содержащий бизнес-логику процессов аутентификации и генерации токенов;
— JwtStrategy — стратегия Passport.js для валидации JWT-токенов;
— AuthGuard — защитный механизм для ограничения доступа к защищённым ресурсам;
— RolesGuard — механизм проверки ролевой принадлежности пользователя;
— DTO-классы — схемы валидации входящих данных.

Процесс аутентификации реализуется следующим образом. При выполнении запроса к эндпоинту POST /api/auth/login система принимает учётные данные пользователя (логин и пароль). AuthService инициирует поиск пользователя по логину через UserService. В случае обнаружения пользователя производится верификация пароля посредством сравнения хеша, хранящегося в базе данных, с хешем введённого пароля. Для хеширования применяется алгоритм Argon2, обладающий устойчивостью к атакам перебора и радужным таблицам.

При успешной верификации генерируется пара токенов: access-token со сроком действия один час и refresh-token со сроком действия семь дней. Access-token содержит минимальный набор утверждений (claims): идентификатор пользователя и его роль. Refresh-token хранится в httpOnly cookie, что предотвращает доступ к токену из JavaScript-кода и снижает риск XSS-атак (Cross-Site Scripting).

Механизм обновления токенов реализован через эндпоинт POST /api/auth/refresh. При истечении срока действия access-token клиентское приложение автоматически направляет запрос с refresh-token. Система выполняет верификацию refresh-token, извлекает идентификатор пользователя и генерирует новую пару токенов.

Защита защищённых ресурсов осуществляется посредством декораторов @Auth() и @Roles(). Декоратор @Auth() активирует JwtAuthGuard, который проверяет наличие и валидность JWT-токена в заголовке Authorization. Декоратор @Roles() указывает требуемые роли для доступа к конкретному эндпоинту. RolesGuard сравнивает роль авторизованного пользователя с перечнем разрешённых ролей и предоставляет или ограничивает доступ соответственно.

Для получения данных текущего пользователя разработаны кастомные декораторы @CurrentUser() и @CurrentUserId(), извлекающие информацию из контекста выполнения запроса. Это позволяет упростить код контроллеров и избежать дублирования логики извлечения данных пользователя.

2.3.3. Реализация модулей бизнес-логики

Модули бизнес-логики составляют функциональное ядро системы и обеспечивают выполнение основных бизнес-процессов салона красоты. В данном разделе рассматриваются ключевые модули: управления мастерами, управления услугами и прайс-листом, управления расписанием, управления записями клиентов.

Модуль управления мастерами (MasterModule)

Модуль предназначен для выполнения CRUD-операций (Create, Read, Update, Delete) над данными мастеров салона. MasterController определяет следующие эндпоинты:
— GET /api/master — получение списка всех мастеров;
— GET /api/master/:id — получение детальной информации о мастере;
— POST /api/master — создание нового мастера (доступно администратору);
— PATCH /api/master/:id — обновление данных мастера;
— DELETE /api/master/:id — деактивация мастера.

MasterService реализует бизнес-логику операций. Особенностью реализации является применение стратегии мягкого удаления (soft delete): при удалении мастера устанавливается флаг isActive = false, что позволяет сохранить целостность исторических данных о записях. При создании мастера с опциональными учётными данными автоматически создаётся связанная учётная запись пользователя с ролью «master».

Модуль управления услугами и прайс-листом (ServiceModule, ServicePriceModule)

ServiceModule управляет справочником услуг салона. Каждая услуга характеризуется наименованием, описанием, продолжительностью, изображением и принадлежностью к категории. Модуль поддерживает фильтрацию услуг по категориям и статусу активности.

ServicePriceModule реализует связь «многие-ко-многим» между мастерами и услугами. Данный модуль позволяет устанавливать индивидуальную стоимость каждой услуги для конкретного мастера, что соответствует бизнес-требованиям салона, где ценообразование может варьироваться в зависимости от квалификации специалиста. Также предусмотрена возможность переопределения стандартной продолжительности услуги для конкретного мастера (поле durationOverride).

Модуль управления расписанием (MasterScheduleModule)

Модуль обеспечивает управление рабочим временем мастеров. Расписание задаётся для каждого дня недели (от 0 до 6, где 0 — воскресенье) и включает время начала и окончания рабочего дня. Хранение расписания в виде отдельных записей для каждого дня недели позволяет гибко настраивать графики работы с различными временными интервалами.

Дополнительно модуль управляет периодами отсутствия мастеров (MasterTimeOff). Период отсутствия характеризуется датами начала и окончания, типом (отпуск, больничный, отгул, иное) и комментарием. При проверке доступности мастера система выполняет последовательную валидацию:
— проверка наличия периода отсутствия в запрашиваемую дату;
— проверка наличия расписания для соответствующего дня недели;
— проверка отсутствия конфликтующих записей на указанное время.

Модуль управления записями клиентов (AppointmentModule)

AppointmentModule представляет собой наиболее сложный функциональный модуль системы, обрабатывающий полный жизненный цикл записи клиента. Модуль поддерживает следующие операции:
— создание записи через публичный портал или административную панель;
— подтверждение, отмена и завершение записей;
— перенос записей на иное время;
— фильтрация записей по мастерам, датам и статусам;
— автоматическое архивирование завершённых записей в таблицу истории.

Ключевым методом сервиса AppointmentService является метод ensureTimeNotTaken, осуществляющий проверку доступности временного слота. Ниже приведён фрагмент реализации данного метода:

```typescript
private async ensureTimeNotTaken(masterId: number, time: string) {
  const existing = await this.prisma.appointment.findFirst({
    where: {
      masterID: masterId,
      appointmentTime: new Date(time),
      status: {
        notIn: [AppointmentStatus.Отменен, AppointmentStatus.Завершен]
      }
    }
  });

  if (existing) {
    throw new BadRequestException(
      `На ${time} мастер уже занят активной записью`
    );
  }
}
```

При создании записи выполняется каскадная валидация:
— существование мастера с указанным идентификатором;
— существование услуги с указанным идентификатором;
— наличие установленной цены на услугу для данного мастера;
— отсутствие конфликтующих записей на указанное время.

Цена услуги фиксируется в момент создания записи в поле price таблицы appointment. Это обеспечивает неизменность стоимости даже при последующем изменении прайс-листа.

Система статусов записей включает четыре значения: «Новый», «Подтверждён», «Завершён», «Отменён». Переход между статусами регламентирован бизнес-логикой. Запись со статусом «Новый» требует подтверждения администратором или мастером. Запись со статусом «Подтверждён» считается активной и отображается в расписании. После оказания услуги запись переводится в статус «Завершён», при этом автоматически создаётся копия в таблице appointment_history для последующего анализа. Записи со статусом «Отменён» исключаются из расписания, но сохраняются в базе для ведения статистики отмен.

Для автоматического завершения просроченных записей реализована задача Cron (appointment-cron.service.ts), выполняющаяся ежечасно. Задача обнаруживает записи со статусом «Подтверждён», время которых истекло, и переводит их в статус «Завершён».

2.4. Разработка клиентской части (Frontend)

2.4.1. Структура клиентского приложения и компоненты интерфейса

Клиентская часть автоматизированной системы разработана с применением платформы Next.js версии 15, функционирующей на базе библиотеки React версии 19. Выбор Next.js обусловлен необходимостью поддержки серверного рендеринга (SSR — Server-Side Rendering) для публичных страниц с целью поисковой оптимизации (SEO), а также встроенными механизмами маршрутизации через файловую систему и оптимизации производительности.

Структура клиентского приложения организована в соответствии с принципом разделения ответственности и включает следующие функциональные зоны:

— Публичный портал (Public Zone) — группа страниц, доступных без авторизации:
  • Главная страница — содержит приветственный блок, информацию о салоне, перечень преимуществ, фотографии интерьера и мастеров;
  • Каталог услуг — страница с фильтрацией услуг по категориям, отображением наименования, описания, продолжительности и стоимости услуг;
  • Форма записи — интерактивный интерфейс для создания записи клиентом, включающий выбор мастера, услуги, даты и времени, а также ввод контактных данных.

— Личный кабинет мастера (Master Zone) — защищённая зона для авторизованных пользователей с ролью «мастер»:
  • Расписание мастера — календарь с отображением записей на неделю, цветовой индикацией статусов записей и быстрыми действиями;
  • Список записей — таблица записей с фильтрацией по статусу и дате, возможностью просмотра деталей, подтверждения, отмены и завершения записей;
  • Профиль мастера — страница с персональными данными мастера, списком оказываемых услуг и установленными ценами.

— Административная панель (Admin Zone) — защищённая зона для пользователей с ролью «администратор»:
  • Управление мастерами — интерфейс для создания, редактирования и деактивации мастеров, включая загрузку фотографий и настройку учётных записей;
  • Управление услугами и категориями — инструменты для ведения справочников услуг и категорий, настройки прайс-листов;
  • Управление расписанием — интерфейс для установки рабочего времени мастеров и управления периодами отсутствия;
  • Управление записями — просмотр всех записей с расширенной фильтрацией, подтверждение новых записей, отмена и перенос;
  • Аналитика — дашборды с графиками и таблицами, отображающими финансовые показатели, популярность услуг, загрузку мастеров и клиентскую аналитику.

Компонентная архитектура приложения построена с использованием современных подходов React:
— Server Components — компоненты, выполняющиеся на сервере и передающие готовый HTML клиенту, что снижает объём загружаемого JavaScript-кода;
— Client Components — интерактивные компоненты с локальным состоянием, использующие хуки React;
— Custom Hooks — пользовательские хуки для инкапсуляции повторяющейся логики работы с данными;
— Service Layer — абстракция для вызовов API, вынесенная в отдельный модуль services.

2.4.2. Взаимодействие с API на клиенте

Взаимодействие клиентской части с серверным API реализовано посредством библиотеки Axios, предоставляющей удобный интерфейс для выполнения HTTP-запросов и обработки ответов. Для каждого функционального модуля создан соответствующий сервис:

— authService — операции аутентификации (вход, выход, обновление токена);
— masterService — CRUD-операции с данными мастеров;
— serviceService — операции с услугами и категориями;
— appointmentService — операции с записями клиентов;
— scheduleService — управление расписанием и периодами отсутствия;
— analyticsService — получение аналитических данных.

Каждый сервис экспортирует методы, соответствующие эндпоинтам API. Например, appointmentService включает методы getAll, getById, create, update, complete, delete, getByDate.

Для управления состоянием сервера применяется библиотека TanStack Query (React Query). Данная библиотека обеспечивает:
— кэширование данных запросов с настраиваемым временем жизни;
— фоновое обновление данных при фокусе окна или возврате к вкладке;
— автоматические повторные запросы при ошибках сети;
— оптимистичные обновления для улучшения пользовательского опыта.

Пример использования TanStack Query для получения списка записей:

```typescript
const { data: appointments, isLoading, error } = useQuery({
  queryKey: ['appointments', date, masterId],
  queryFn: () => appointmentService.getByDate(date, masterId),
  staleTime: 1000 * 60 * 5 // Данные считаются актуальными 5 минут
});
```

Для обработки форм применяется библиотека React Hook Form в связке с валидатором Zod. React Hook Form минимизирует количество ре-рендеров за счёт управления состоянием формы вне React-состояния. Zod используется для декларативного описания схем валидации полей формы.

Межсервисная коммуникация реализована через аксиос-инстансы с настроенными интерсепторами (interceptors). Интерсептор запросов автоматически добавляет access-token к заголовкам защищённых запросов. Интерсептор ответов обрабатывает ошибки аутентификации (статус 401) и инициирует автоматическое обновление токена.

2.4.3. Адаптивность и пользовательский опыт

Разработанный интерфейс адаптирован для отображения на устройствах с различными размерами экранов: настольных компьютерах, планшетах и мобильных телефонах. Адаптивность реализована с помощью утилитарных классов Tailwind CSS с префиксами bps (breakpoints):

— sm (small) — устройства шириной от 640 пикселей;
— md (medium) — устройства шириной от 768 пикселей;
— lg (large) — устройства шириной от 1024 пикселей;
— xl (extra large) — устройства шириной от 1280 пикселей.

Для каждого компонента определены стили для различных точек останова. Например, таблица записей на настольных компьютерах отображается в виде полной таблицы со всеми колонками, на планшетах — с сокращённым набором колонок, на мобильных устройствах — в виде вертикальных карточек.

Пользовательский опыт (UX) улучшен за счёт применения следующих решений:
— индикаторы загрузки (скелетоны и спиннеры) для визуального информирования пользователя о процессе получения данных;
— всплывающие уведомления (toast) для отображения результатов операций (успех, ошибка, предупреждение);
— анимации переходов между страницами и состояниями с использованием библиотеки Framer Motion;
— валидация форм в реальном времени с отображением сообщений об ошибках до отправки данных;
— подтверждение критических действий (удаление, отмена записи) через модальные диалоги.

Цветовая схема интерфейса соответствует фирменному стилю салона красоты «ЭДЭН» и включает следующие основные цвета:
— основной акцентный цвет — #7C3AED (фиолетовый);
— цвет успеха — #10B981 (зелёный);
— цвет предупреждения — #F59E0B (жёлтый);
