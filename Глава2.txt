ГЛАВА 2. ПРОЕКТИРОВАНИЕ И РАЗРАБОТКА АВТОМАТИЗИРОВАННОЙ СИСТЕМЫ

2.1. Проектирование базы данных

2.1.1. Концептуальная модель данных

Проектирование базы данных является фундаментальным этапом создания автоматизированной системы управления салоном красоты. На основе анализа бизнес-процессов ИП «Бокова Э.А.» была разработана концептуальная модель данных. Модель отражает основные сущности и их взаимосвязи в системе.

Концептуальная модель включает следующие основные сущности:

1. Мастер (Master) — сотрудник салона, оказывающий услуги. Сущность содержит информацию о фамилии, имени, отчестве мастера. Также содержит данные о специализации, контактном номере телефона, статусе активности и фотографии.

2. Пользователь (User) — учетная запись для авторизации в системе. Каждый мастер может иметь связанную учетную запись пользователя для входа в системе. Сущность содержит имя пользователя (логин), хэш пароля, роль (admin или master) и ссылку на мастера.

3. Услуга (Service) — услуга, оказываемая в салоне. Сущность содержит название услуги, описание, продолжительность в минутах, изображение и категорию, к которой относится услуга.

4. Категория услуг (Category) — группа услуг по типу. Категория объединяет услуги по их направленности. Примеры категорий: «Стрижка», «Окрашивание», «Маникюр».

5. Прайс-лист услуг (ServicePrice) — стоимость конкретной услуги у конкретного мастера. Данная сущность реализует связь «многие-ко-многим» между мастерами и услугами. Это позволяет каждому мастеру устанавливать свою цену на каждую услугу. Также предусмотрена возможность переопределения длительности услуги для конкретного мастера.

6. Расписание мастера (MasterSchedule) — рабочее время мастера по дням недели. Сущность определяет начало и конец рабочего дня для каждого мастера в каждый день недели.

7. Отсутствие мастера (MasterTimeOff) — период отсутствия мастера на работе. Это может быть отпуск, больничный или выходной. Сущность содержит даты начала и окончания отсутствия, тип причины и комментарий.

8. Запись на прием (Appointment) — запись клиента на оказание услуги. Сущность содержит фамилию, имя и телефон кли��нта. Также содержит ссылку на мастера, ссылку на услугу, дату и время записи, цену и статус записи.

9. История записей (AppointmentHistory) — архив завершенных записей. Сущность используется для хранения истории завершенных записей. Необходима для аналитики и статистики.

Взаимосвязи между сущностями:

- Один мастер может иметь одну учетную запись пользователя (связь 1:1).
- Один мастер может иметь множество записей на прием (связь 1:M).
- Один мастер может иметь множество записей в прайс-листе (связь 1:M).
- Один мастер может иметь множество записей расписания (связь 1:M).
- Один мастер может иметь множество периодов отсутствия (связь 1:M).
- Одна категория может содержать множество услуг (связь 1:M).
- Одна услуга может входить в множество записей прайс-листа (связь 1:M).
- Одна услуга может иметь множество записей на прием (связь 1:M).

2.1.3. Физическая модель данных и описание таблиц

Физическая модель данных определяет способ хранения данных в конкретной системе управления базой данных. В проекте используется PostgreSQL в связке с ORM Prisma. Prisma обеспечивает генерацию SQL-запросов и миграцию схемы базы данных.

Для обеспечения оптимальной производительности и целостности данных определены индексы:

Индексы таблицы «master_schedule»:

- Индекс по полю masterID. Используется для быстрого поиска расписания конкретного мастера.
- Составной индекс по полям (masterID, dayOfWeek). Используется для поиска расписания мастера в определенный день.
- Индекс по полям (startTime, endTime). Используется для проверки доступного времени.

Связь между таблицами реализуется через внешние ключи с каскадным удалением:

- При удалении мастера удаляются все его записи расписания (onDelete: Cascade).
- При удалении категории автоматически удаляются связанные услуги.

Типы данных PostgreSQL используются с учетом требований:

- DECIMAL(10,2) для денежных сумм. Тип обеспечивает точность вычислений без погрешностей floating-point.
- TIMESTAMP для дат и времени. Тип позволяет хранить дату и время с точностью до микросекунды.
- ENUM для статусов и ролей. Тип обеспечивает целостность данных на уровне СУБД.

Нормализация базы данных:

- Все таблицы приведены к третьей нормальной форме (3NF).
- Устранены транзитивные зависимости.
- Каждая таблица имеет первичный ключ.
- Использованы уникальные ограничения для полей, которые не должны дублироваться.

2.2. Архитектура автоматизированной системы

2.2.1. Общая архитектура приложения

Разработанная автоматизированная система представляет собой трехуровневую клиент-серверную архитектуру:

1. Уровень представления (Presentation Layer) — клиентская часть на Next.js 15. Часть работает в браузере пользователя. Уровень отвечает за отображение пользовательского интерфейса. Также отвечает за обработку действий пользователя и взаимодействие с сервером через HTTP-запросы.

2. Уровень бизнес-логики (Business Logic Layer) — серверная часть на NestJS. Часть развернута как REST API. Уровень обрабатывает запросы от клиента. Выполняет бизнес-логику. Взаимодействует с базой данных. Возвращает результаты.

3. Уровень данных (Data Layer) — система управления базой данных PostgreSQL с ORM Prisma. Уровень обеспечивает хранение, извлечение и модификацию данных.

Клиент-серверное взаимодействие осуществляется по протоколу HTTP. Используется формат JSON для обмена данными. Авторизация реализована с использованием JWT-токенов (JSON Web Token). Токены передаются в заголовках HTTP-запросов.

2.2.2. Выбор паттерна проектирования

Для серверной части выбран модульный паттерн проектирования. Паттерн предоставляется фреймворком NestJS. Каждый функциональный модуль системы является отдельным модулем NestJS со своей структурой:

- Module — файл определения модуля, импортирующий зависимости.
- Controller — файл обработки HTTP-запросов (эндпоинты API).
- Service — файл бизнес-логики.
- DTO (Data Transfer Object) — файлы определения структур данных для валидации.

Для клиентской части используется паттерн компонентной архитектуры React. Используются современные подходы:

- Server Components (Next.js App Router) — компоненты, работающие на сервере.
- Client Components — интерактивные компоненты с состоянием.
- Custom Hooks — инкапсуляция логики работы с данными.
- Service Layer — абстракция API-вызовов.

2.2.3. Структура клиент-серверного взаимодействия

Клиентская часть разделена на три логические зоны:

1. Публичный портал (Public Zone) — страницы, доступные без авторизации:
   - Главная страница салона.
   - Каталог услуг с категориями.
   - Форма записи на прием.

2. Личный кабинет мастера (Master Zone) — страницы для мастеров:
   - Расписание мастера.
   - Список записей.
   - Профи��ь мастера.

3. Административная панель (Admin Zone) — страницы для администраторов:
   - Управление мастерами.
   - Управление услугами и категориями.
   - Управление расписанием.
   - Просмотр и подтверждение записей.
   - Аналитика и статистика.

Серверные эндпоинты также разделены по зонам. Применяются JWT-guards для авторизации. Применяются role-guards для проверки прав доступа.

2.3. Проектирование API системы

2.3.1. Структура REST API

REST API разработано в соответствии с принципами REST (Representational State Transfer). Используется принцип разделения на ресурсы:

- /api/auth — аутентификация и авторизация.
- /api/master — управление мастерами.
- /api/user — управление пользователями.
- /api/category — управление категориями услуг.
- /api/service — управление услугами.
- /api/service-price — управление прайс-листом.
- /api/master-schedule — управление расписанием мастеров.
- /api/appointment — управление записями на прием.
- /api/appointment-history — история записей.
- /api/stats — статистика.
- /api/analytics — аналитика.

2.3.2. Аутентификация и авторизация

Система аутентификации реализована с использованием JWT (JSON Web Tokens):

1. При входе пользователя (POST /api/auth/login) сервер проверяет учетные данные. Сервер возвращает JWT-токен доступа и refresh-токен в httpOnly cookies.

2. JWT-токен содержит информацию о пользователе (ID, логин, роль). Токен имеет срок действия.

3. При каждом защищенном запросе токен проверяется JWT-guard.

4. Role-guard проверяет роль пользователя для доступа к защищенным ресурсам.

5. Для мастеров дополнительно проверяется связь с учетной записью пользователя.

Модуль мастеров (MasterModule) обеспечивает CRUD-операции над данными мастеров. Основные функции:

- Получение списка активных мастеров с информацией об оказываемых услугах.
- Получение детальной информации о мастере (включая специализацию и расписание).
- Создание нового мастера с автоматическим созданием связанной учетной записи.
- Обновление информации о мастере.
- Деактивация мастера (мягкое удаление с сохранением данных).
- Получение списка услуг мастера с ценами.

Реализация включает:

- MasterController — эндпоинты API.
- MasterService — бизнес-логика.

- MasterModule — организация модуля.

2.4.3. Реализация модуля услуг и прайс-листа

Модуль услуг (ServiceModule) и прайс-листа (ServicePriceModule) обеспечивают управление услугами салона:

Функции модуля услуг:

- Получение детальной информации об услуге.
- Создание новой услуги с привязкой к категории.
- Обновление информации об услуге.
- Деактивация услуги.

Функции модуля прайс-листа:

- Получение прайс-листа для конкретного мастера.
- Установка/обновление цены на услугу для мастера.
- Установка переопределенной длительности услуги для мастера.
- Деактивация записи прайс-листа.
- Массовое обновление прайс-листа.

2.4.4. Реализация модуля расписания

Модуль расписания (MasterScheduleModule) управляет рабочим временем мастеров:

- Получение расписания мастера на неделю.
- Установка рабочего времени мастера по дням недели.
- Управление периодами отсутствия (отпуск, больничный, выходные).
- Проверка доступности мастера на указанное время.
- Получение слотов доступного времени для записи.

Особенности реализации:

- Расписание хранится для каждого дня недели отдельно.
- Периоды отсутствия имеют приоритет над обычным расписанием.
- Автоматическая проверка конфликтов при создании записи.

2.4.5. Реализация модуля записей клиентов

Модуль записей (AppointmentModule) обеспечивает работу с записями клиентов:

- Создание новой записи клиентом через публичный портал.
- Подтверждение записи администратором.
- Отмена записи (клиентом или администратором).
- Завершение записи (после оказания услуги).
- Перенос записи на другое время.
- История записей с возможностью фильтрации.

Особенности реализации:

- При создании записи автоматически проверяется доступность мастера.
- Цена фиксируется на момент создания записи.
- При завершении записи автоматически создается запись в истории.
- Автоматический cron-задача для завершения просроченных записей.

2.4.6. Реализация модуля аналитики

Модуль аналитики (AnalyticsModule) предоставляет статистическую информацию:

Финансовая аналитика:

- Выручка за период (день, неделя, месяц, год).
- Средний чек.
- Количество транзакций.
- График изменения выручки по дням.

Аналитика по услугам:

- Популярность услуг (количество записей).
- Выручка по услугам.
- Средняя загрузка услуг.

Аналитика по мастерам:

- Загрузка мастера (количество записей).
- Выручка мастера.
- Рейтинг мастеров по эффективности.

Аналитика по клиентам:

- Новые и повторные клиенты.
- Частота посещений.
- Лояльность клиентов.

Сравнительный анализ:

- Сравнение показателей по периодам.
- Сравнение мастеров между собой.
- Динамика изменения показателей.

2.5. Разработка клиентской части (Frontend)

2.5.1. Настройка Next.js и структура проекта

Клиентская часть разработана на Next.js 15 с использованием React 19. Структура проекта:

Frontend/
├── src/
│   ├── app/ — приложение (App Router).
│   │   ├── (public)/ — публичные страницы.
│   │   ├── (master)/ — страницы мастера.
│   │   ├── (admin)/ — страницы администратора.
│   │   ├── (auth)/ — страницы авторизации.

2.7. Внедрение и эксплуатация системы

2.7.1. Требования к развертыванию

Для развертывания системы требуются следующие аппаратные и программные средства:

Серверная часть:

- Операционная система: Linux (Ubuntu 20.04+) или Windows Server.
- Node.js версии 18+.
- PostgreSQL версии 14+.
- Nginx для проксирования (рекомендуется).
- Минимум 2 ГБ оперативной памяти.
- Минимум 20 ГБ дискового пространства.

Клиентская часть:

- Современный веб-браузер (Chrome 90+, Firefox 90+, Safari 15+, Edge 90+).
- Подключение к интернету.
- JavaScript включен.

Рекомендуемые параметры сервера для production:

- Оперативная память: 4+ ГБ.
- CPU: 2+ ядра.
- SSD-накопитель для базы данных.

2.7.2. Порядок развертывания системы

Процесс развертывания включает следующие этапы:

1. Подготовка сервера:
   - Установка Node.js.
   - Установка PostgreSQL.
   - Настройка сетевого подключения.

2. Развертывание базы данных:
   - Создание базы данных.
   - Выполнение миграций Prisma.
   - Заполнение начальных данных (seeds).

3. Развертывание Backend:
   - Клонирование репозитория.
   - Установка зависимостей (npm install).
   - Настройка переменных окружения.
   - Сборка проекта (npm run build).
   - Запуск приложения.

4. Развертывание Frontend:
   - Клонирование репозитория.
   - Установка зависимостей.
   - Настройка переменных окружения.
   - Сборка проекта (npm run build).
   - Запуск или настройка Nginx для статических файлов.

5. Настройка обратного прокси (Nginx):
   - Настройка проксирования на Backend.
   - Настройка раздачи статических файлов Frontend.
   - Настройка HTTPS (SSL-сертификат).
   - Настройка кэширования.

6. Проверка работоспособности:
   - Проверка доступности API.
   - Проверка доступности клиента.
   - Проверка авторизации.
   - Проверка создания тестовой записи.

7. Запуск в production режиме:
   - Настройка PM2 для автозапуска.
   - Настройка мониторинга.
   - Настройка резервного копирования базы данных.
